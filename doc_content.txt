Unbiased AI Decision
Ensuring Fairness & Detecting Bias in Automated Decisions
Project Documentation  ·  8-Day Sprint  ·  2-Person Team
1. Project Overview
Computer programs now make life-changing decisions — who gets hired, who receives a bank loan, who qualifies for medical care. When these systems learn from flawed or historically biased data, they perpetuate and amplify those exact injustices at scale.

Unbiased AI Decision is a full-stack web platform that empowers data teams and organisations to audit their datasets and ML models for hidden bias. It surfaces bias metrics, provides explainability insights, and generates downloadable audit reports — making fairness accessible to every organisation regardless of their AI expertise.

1.1 Objectives
Provide an intuitive upload interface for datasets (CSV/JSON) and trained ML models
Automatically compute industry-standard fairness metrics across sensitive attributes
Generate clear, downloadable bias audit reports with actionable recommendations
Enable organisations to detect and fix harmful bias before systems impact real people

7. API Endpoints Reference
All endpoints are prefixed with /api/v1. Authentication is required for all endpoints except /auth/register and /auth/login. The JWT token must be sent in the Authorization: Bearer <token> header.


8. Bias Detection Metrics
The bias engine computes the following fairness metrics for each pair of demographic groups across every sensitive attribute specified by the user (e.g. gender, race, age bracket):


8.1 Overall Bias Score
Each job produces a single Bias Score (0–100) computed as a weighted average of all metric pass/fail results. Scores are colour-coded:

80–100  → LOW RISK  (green)   — Model meets fairness thresholds
50–79   → MEDIUM RISK  (amber) — Some metrics fail; review recommended
0–49    → HIGH RISK  (red)     — Multiple failures; model should not be deployed

9. UI Screens Specification
9.1 Screen Inventory
Login / Register — Email + password auth, JWT stored in memory
Dashboard — Overview cards: total analyses, avg bias score, flagged models count
Upload Dataset — Drag-and-drop zone, CSV/JSON support, column type preview
Configure Analysis — Select target column, sensitive attribute columns, upload optional model
Analysis Results — Bias score gauge, per-metric breakdown table, SHAP chart
Reports List — Table of all reports with status, score badge, download PDF button
Report Detail — Full metrics, recommendations list, exportable
Settings / Audit Log — User profile, role badge, paginated audit events

9.2 Bias Score Gauge Component
The score gauge is the centrepiece of the Analysis Results screen. It is a semicircular arc built with SVG + Recharts, coloured red→amber→green (0→50→80→100). The numerical score animates in on load. Below it, each metric renders as a row with a coloured pill (PASS / FAIL) and the computed value.

9.3 SHAP Feature Importance Chart
A horizontal bar chart sorted by absolute mean SHAP value. Sensitive attributes are highlighted in amber. If a sensitive attribute ranks in the top 3 features, a warning banner fires: 'Sensitive attribute is a top predictor — consider removing or transforming it.'

10. Environment Setup
10.1 Prerequisites
Python 3.11+,  Node.js 20+,  Docker Desktop,  PostgreSQL 15+,  Redis 7+

10.2 Backend Setup
git clone <repo> && cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # fill in DATABASE_URL, JWT_SECRET, REDIS_URL
alembic upgrade head   # run migrations
uvicorn app.main:app --reload --port 8000
celery -A app.workers.tasks worker --loglevel=info   # in second terminal

10.3 Frontend Setup
cd frontend
npm install
cp .env.example .env   # set VITE_API_URL=http://localhost:8000/api/v1
npm run dev            # starts on http://localhost:5173

10.4 Docker (Recommended)
docker-compose up --build
This spins up: backend (port 8000), frontend (port 5173), PostgreSQL (5432), Redis (6379).

13. Risk Register
Performance — Bias computation is slow for large datasets
Mitigation: Run all analysis in Celery background workers. Cap dataset size at 500K rows initially. Add chunked processing in Week 2 if needed.

Fairlearn / AIF360 API changes
Mitigation: Pin library versions in requirements.txt. Abstract the metrics layer behind a service interface so the underlying library can be swapped.

CORS / API connectivity issues
Mitigation: Configure FastAPI CORS middleware on Day 1 to allow the frontend origin. Use a .env variable so the origin is configurable per environment.

Scope creep on Day 6+
Mitigation: Feature freeze after Day 6. Any new feature requests go into a backlog and are not worked on until the core system is deployed and stable.

14. Glossary
Sensitive Attribute — A feature that defines a demographic group (e.g. gender, race, age)
Target Column — The outcome the ML model predicts (e.g. loan_approved, hired)
Disparate Impact — A legal fairness doctrine; a 4/5ths rule from US employment law
SHAP — SHapley Additive exPlanations; a game-theoretic approach to model explainability
Fairlearn — A Microsoft open-source library for ML fairness assessment
AIF360 — IBM's AI Fairness 360 toolkit with additional bias metrics and algorithms
Celery — A distributed task queue for Python, used here to run async bias jobs
JWT — JSON Web Token; a compact, URL-safe method for representing authentication claims
